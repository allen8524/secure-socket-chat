# Threat Model

## 1. 문서 목적

이 문서는 SecureSocketChat의 보안 설계 관점을 설명하기 위한 위협 모델(threat model) 문서입니다. 현재 구현이 어떤 자산을 보호하려고 하는지, 어떤 신뢰 경계를 두고 있는지, 주요 공격 시나리오에 대해 어떤 방어를 적용했는지, 그리고 아직 남아 있는 제한 사항을 면접과 포트폴리오 설명에 적합한 수준으로 정리합니다.

SecureSocketChat은 운영 환경용 보안 메신저가 아니라, Python socket 기반 채팅 프로그램에 클라이언트-서버 간 암호화 채널과 자체 패킷 프로토콜을 적용한 학습 및 포트폴리오 목적의 보안 네트워크 프로젝트입니다.

## 2. 보호 대상

| 보호 대상 | 설명 | 보호 관점 |
|---|---|---|
| 채팅 메시지 본문 | 전체 채팅과 1:1 귓속말로 전송되는 텍스트 데이터 | 네트워크 도청 시 평문 노출 방지 |
| 이미지 binary payload | 이미지 파일 전송 시 포함되는 바이너리 데이터 | 전송 중 도청 및 변조 가능성 축소 |
| 세션 공개키 fingerprint | 클라이언트와 서버의 공개키 식별 정보 | 현재 세션의 암호화 채널 식별 |
| 사용자명 및 접속 상태 | 접속자 목록, 중복 사용자명, online/offline 상태 | 라우팅 대상 식별 및 접속 상태 관리 |
| 파일명 및 수신 파일 저장 경로 | 수신 이미지 저장 시 사용하는 파일명과 경로 | 경로 조작(path traversal) 가능성 축소 |
| 패킷 header와 payload size 정보 | JSON header, binary payload 크기, length prefix | 비정상 패킷과 과도한 메모리 사용 방어 |

## 3. 신뢰 경계

| 경계 | 신뢰 수준 | 설명 |
|---|---|---|
| 클라이언트 내부 | 부분 신뢰 | 사용자가 실행하는 GUI 클라이언트이며, 입력값과 파일 선택은 검증 대상입니다. |
| 네트워크 구간 | 비신뢰 | 패킷 도청, 변조, 재전송 가능성이 있는 구간으로 간주합니다. |
| 서버 프로세스 | 신뢰 대상 | 현재 구조에서는 서버가 메시지 라우팅을 위해 각 클라이언트와의 암호화 채널에서 메시지를 복호화합니다. |
| 다른 클라이언트 | 비신뢰 | 다른 사용자는 임의의 메시지, 파일명, payload를 보낼 수 있는 주체로 봅니다. |
| 로컬 파일 시스템 | 부분 신뢰 | 수신 이미지 저장 경로는 애플리케이션이 제어하지만, 파일명은 외부 입력에서 올 수 있습니다. |

## 4. 주요 공격 시나리오

| 공격 시나리오 | 영향 | 현재 위험 수준 |
|---|---|---|
| 네트워크 패킷 도청 | 메시지 본문과 이미지 payload가 노출될 수 있음 | 암호화 채널로 완화 |
| 암호문 또는 payload 변조 | 복호화 실패, 잘못된 메시지 처리, 이미지 무결성 훼손 | PyNaCl Box와 SHA-256 검증으로 완화 |
| 비정상 header size 전송 | 과도한 메모리 사용 또는 파싱 오류 유발 | header size 제한으로 완화 |
| payload size 불일치 | 패킷 파싱 흐름 교란 또는 수신 처리 오류 | 프로토콜 검증으로 완화 |
| payload size 초과 전송 | 메모리 사용량 증가 또는 서비스 안정성 저하 | payload size 제한으로 완화 |
| 이미지 파일명 경로 조작 | 의도하지 않은 경로에 파일 저장 시도 | 파일명 정규화로 완화 |
| 중복 사용자명 접속 | 라우팅 대상 혼동, 귓속말 대상 충돌 | 중복 사용자명 차단으로 완화 |
| 서버 위장 가능성 | 사용자가 공격자 서버에 접속할 수 있음 | 현재는 fingerprint 표시만 제공 |
| replay attack 가능성 | 이전 암호문 또는 요청이 재전송될 수 있음 | sequence number 기반 방어는 미적용 |
| 서버 중계 구조의 한계 | 서버가 라우팅 중 메시지를 복호화할 수 있음 | 설계상 제한 사항으로 명시 |

## 5. 현재 대응 방식

| 대응 방식 | 적용 대상 | 설명 |
|---|---|---|
| PyNaCl 기반 공개키 교환 | 암호화 세션 구성 | 클라이언트와 서버가 공개키를 교환한 뒤 PyNaCl Box 기반 채널을 구성합니다. |
| 클라이언트-서버 간 암호화 채널 | 메시지, header, 이미지 payload | 네트워크 구간에서 평문 패킷이 직접 노출되지 않도록 합니다. |
| 4-byte length prefix + JSON header + binary payload | 자체 패킷 프로토콜 | 패킷 길이와 메타데이터, 바이너리 payload를 분리해 framing합니다. |
| header size 제한 | JSON header | 비정상적으로 큰 header로 인한 메모리 사용량 증가를 줄입니다. |
| payload size 제한 | 메시지 및 이미지 payload | 과도한 payload 전송으로 인한 자원 사용과 처리 오류를 줄입니다. |
| SHA-256 기반 이미지 무결성 검증 | 이미지 binary payload | 전송된 이미지 payload가 header의 해시와 일치하는지 확인합니다. |
| 파일명 정규화 | 수신 이미지 파일명 | `os.path.basename()` 기반 처리로 디렉터리 경로 조작 가능성을 줄입니다. |
| 중복 사용자명 차단 | 사용자명 및 접속 상태 | 동일 이름 사용으로 인한 라우팅 혼동을 방지합니다. |
| 보안 세션 정보 및 fingerprint 표시 | 사용자 확인 | GUI와 명령어를 통해 cipher, session ID, 공개키 fingerprint를 확인할 수 있게 합니다. |
| pytest 기반 프로토콜 검증 | 패킷 처리 로직 | 패킷 packing/unpacking, 크기 제한, 잘못된 패킷 방어를 테스트합니다. |

## 6. 남아 있는 제한 사항

| 제한 사항 | 설명 |
|---|---|
| 종단 간 암호화 미보장 | 현재 구조는 클라이언트-서버 간 암호화 채널입니다. 서버는 메시지 라우팅을 위해 각 클라이언트와의 암호화 채널에서 메시지를 복호화합니다. |
| 서버 인증 부재 | 인증기관(CA) 기반 서버 인증은 없습니다. 사용자는 접속한 서버가 기대한 서버인지 자동으로 검증하지 않습니다. |
| TOFU/키 핀닝 미적용 | Trust On First Use 방식의 fingerprint 저장이나 키 변경 감지는 아직 적용되지 않았습니다. |
| replay 방어 미적용 | sequence number 또는 nonce 추적 기반의 replay attack 방어는 아직 적용되지 않았습니다. |
| 사용자 인증 부재 | 사용자명은 라우팅 식별자 역할을 하지만, 계정 인증이나 권한 검증 체계는 없습니다. |
| 운영 환경 보안 한계 | 이 프로젝트는 운영 환경용 보안 메신저가 아니라 학습 및 포트폴리오 목적의 보안 네트워크 프로젝트입니다. |

## 7. 향후 개선 방향

| 개선 방향 | 기대 효과 |
|---|---|
| TOFU 기반 서버 fingerprint 저장 및 변경 감지 | 최초 접속 서버의 fingerprint를 저장하고 이후 변경 여부를 사용자에게 경고할 수 있습니다. |
| sequence number 기반 replay 방어 | 메시지 순서와 재전송 여부를 추적해 이전 패킷 재사용 가능성을 줄일 수 있습니다. |
| 실험적 E2E 모드 | 서버가 라우팅만 수행하고 클라이언트 간 메시지 내용을 복호화하지 않는 구조를 실험할 수 있습니다. |
| 파일 전송 확장 | 이미지 외 파일 타입, 파일 크기 정책, 확장자 검증, 저장 정책을 체계화할 수 있습니다. |
| 통합 테스트 확대 | 실제 서버와 다중 클라이언트 흐름을 포함한 end-to-end 수준의 테스트를 추가할 수 있습니다. |
| 보안 로그와 패킷 검사 도구 추가 | 비정상 패킷, 실패한 복호화, 크기 제한 위반 등을 분석하기 쉽게 만들 수 있습니다. |

## 8. 면접 설명 포인트

| 질문 방향 | 설명 포인트 |
|---|---|
| 왜 자체 패킷 프로토콜을 만들었는가 | 텍스트 메시지와 이미지 payload를 같은 네트워크 흐름에서 다루기 위해 length prefix, JSON header, binary payload를 분리했습니다. |
| 어떤 보안 문제를 고려했는가 | 네트워크 도청, payload 변조, 비정상 크기 패킷, 파일명 경로 조작, 중복 사용자명으로 인한 라우팅 혼동을 고려했습니다. |
| 암호화 구조의 한계는 무엇인가 | 서버가 라우팅을 위해 메시지를 복호화하므로, 현재 구조는 클라이언트-서버 간 암호화 채널이며 서버를 신뢰 경계 안에 둡니다. |
| fingerprint 표시는 왜 필요한가 | 현재 세션에서 사용 중인 공개키 식별 정보를 사용자에게 보여주어 세션 상태를 설명하고 디버깅할 수 있게 합니다. |
| 테스트는 무엇을 검증하는가 | 패킷 packing/unpacking, header와 payload 크기 제한, 잘못된 패킷 방어, 암호화/복호화, SHA-256 해시 검증을 확인합니다. |
| 다음 보안 개선은 무엇인가 | TOFU 기반 서버 fingerprint 저장, sequence number 기반 replay 방어, 실험적 E2E 모드, 통합 테스트 확대를 우선순위로 설명할 수 있습니다. |
