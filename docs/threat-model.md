# Threat Model

## 1. 문서 목적

이 문서는 SecureSocketChat의 보안 설계 관점을 설명하기 위한 위협 모델(threat model) 문서입니다. 현재 구현이 어떤 자산을 보호하려고 하는지, 어떤 신뢰 경계를 두고 있는지, 주요 공격 시나리오에 대해 어떤 방어를 적용했는지, 그리고 아직 남아 있는 제한 사항을 면접과 포트폴리오 설명에 적합한 수준으로 정리합니다.

SecureSocketChat은 운영 환경용 보안 메신저가 아니라, Python socket 기반 채팅 프로그램에 클라이언트-서버 간 암호화 채널과 자체 패킷 프로토콜을 적용한 학습 및 포트폴리오 목적의 보안 네트워크 프로젝트입니다.

## 2. 보호 대상

| 보호 대상 | 설명 | 보호 관점 |
|---|---|---|
| 채팅 메시지 본문 | 전체 채팅과 1:1 귓속말로 전송되는 텍스트 데이터 | 네트워크 도청 시 평문 노출 방지 |
| 파일 binary payload | 이미지와 일반 파일 전송 시 포함되는 바이너리 데이터 | 전송 중 도청 및 변조 가능성 축소 |
| E2E whisper 본문 | `/e2e` 명령으로 보내는 1:1 텍스트 whisper 본문 | 서버가 라우팅 중 본문을 복호화하지 못하도록 보호 |
| 세션 공개키 fingerprint | 클라이언트와 서버의 공개키 식별 정보 | 현재 세션의 암호화 채널 식별 |
| 사용자명 및 접속 상태 | 접속자 목록, 중복 사용자명, online/offline 상태 | 라우팅 대상 식별 및 접속 상태 관리 |
| 파일명 및 수신 파일 저장 경로 | 수신 이미지/파일 저장 시 사용하는 파일명과 경로 | 경로 조작(path traversal) 가능성 축소 |
| 패킷 header와 payload size 정보 | JSON header, binary payload 크기, length prefix | 비정상 패킷과 과도한 메모리 사용 방어 |

## 3. 신뢰 경계

| 경계 | 신뢰 수준 | 설명 |
|---|---|---|
| 클라이언트 내부 | 부분 신뢰 | 사용자가 실행하는 GUI 클라이언트이며, 입력값과 파일 선택은 검증 대상입니다. |
| 네트워크 구간 | 비신뢰 | 패킷 도청, 변조, 재전송 가능성이 있는 구간으로 간주합니다. |
| 서버 프로세스 | 신뢰 대상 | 현재 구조에서는 서버가 메시지 라우팅을 위해 각 클라이언트와의 암호화 채널에서 메시지를 복호화합니다. |
| 다른 클라이언트 | 비신뢰 | 다른 사용자는 임의의 메시지, 파일명, payload를 보낼 수 있는 주체로 봅니다. |
| 로컬 파일 시스템 | 부분 신뢰 | 수신 이미지/파일 저장 경로는 애플리케이션이 제어하지만, 파일명은 외부 입력에서 올 수 있습니다. |

## 4. 주요 공격 시나리오

| 공격 시나리오 | 영향 | 현재 위험 수준 |
|---|---|---|
| 네트워크 패킷 도청 | 메시지 본문과 파일 payload가 노출될 수 있음 | 암호화 채널로 완화 |
| 암호문 또는 payload 변조 | 복호화 실패, 잘못된 메시지 처리, 파일 무결성 훼손 | PyNaCl Box와 SHA-256 검증으로 완화 |
| 비정상 header size 전송 | 과도한 메모리 사용 또는 파싱 오류 유발 | header size 제한으로 완화 |
| payload size 불일치 | 패킷 파싱 흐름 교란 또는 수신 처리 오류 | 프로토콜 검증으로 완화 |
| payload size 초과 전송 | 메모리 사용량 증가 또는 서비스 안정성 저하 | payload size 제한으로 완화 |
| 파일명 경로 조작 | 의도하지 않은 경로에 파일 저장 시도 | 파일명 정규화로 완화 |
| 위험 확장자 파일 전송 | 사용자가 실행 파일이나 스크립트를 실행할 수 있음 | 송신 전 GUI 경고로 완화 |
| 중복 사용자명 접속 | 라우팅 대상 혼동, 귓속말 대상 충돌 | 중복 사용자명 차단으로 완화 |
| 서버 위장 가능성 | 사용자가 공격자 서버에 접속할 수 있음 | TOFU 기반 fingerprint 저장 및 변경 경고로 완화 |
| replay attack 가능성 | 이전 암호문 또는 요청이 재전송될 수 있음 | 채널 단위 sequence number 검증으로 완화 |
| 서버 중계 구조의 한계 | 서버가 라우팅 중 메시지를 복호화할 수 있음 | 설계상 제한 사항으로 명시 |
| E2E key substitution | 서버가 사용자 E2E public key를 바꿔치기할 수 있음 | 현재는 제한 사항으로 남아 있음 |

## 5. 현재 대응 방식

| 대응 방식 | 적용 대상 | 설명 |
|---|---|---|
| PyNaCl 기반 공개키 교환 | 암호화 세션 구성 | 클라이언트와 서버가 공개키를 교환한 뒤 PyNaCl Box 기반 채널을 구성합니다. |
| 클라이언트-서버 간 암호화 채널 | 메시지, header, 파일 payload | 네트워크 구간에서 평문 패킷이 직접 노출되지 않도록 합니다. |
| TOFU 기반 서버 fingerprint 검증 | 서버 위장 가능성 인지 | 최초 접속한 host:port의 서버 fingerprint를 로컬에 저장하고, 이후 변경되면 GUI 경고를 표시합니다. |
| 4-byte length prefix + JSON header + binary payload | 자체 패킷 프로토콜 | 패킷 길이와 메타데이터, 바이너리 payload를 분리해 framing합니다. |
| header size 제한 | JSON header | 비정상적으로 큰 header로 인한 메모리 사용량 증가를 줄입니다. |
| payload size 제한 | 메시지 및 파일 payload | 과도한 payload 전송으로 인한 자원 사용과 처리 오류를 줄입니다. |
| SHA-256 기반 파일 무결성 검증 | 이미지/파일 binary payload | 전송된 payload가 header의 해시와 일치하는지 확인합니다. |
| sequence number 기반 replay 방어 | 암호화 logical packet | 각 SecureChannel이 송신 sequence를 증가시키고, 수신 측은 마지막으로 처리한 sequence 이하의 패킷을 차단합니다. |
| 실험적 E2E whisper mode | 1:1 텍스트 whisper 본문 | 클라이언트가 수신자 E2E public key로 inner payload를 암호화해 서버가 본문을 복호화하지 못하게 합니다. |
| E2E fingerprint 확인 UX | E2E public key 식별 | `/fingerprint 사용자명`과 GUI 버튼으로 상대방의 현재 세션 E2E fingerprint를 확인할 수 있게 합니다. |
| 파일명 정규화 | 수신 이미지/파일명 | 경로 요소 제거와 구분자 치환으로 디렉터리 경로 조작 가능성을 줄입니다. |
| 위험 확장자 경고 | 송신 파일 선택 | 실행 파일 또는 스크립트로 보이는 파일을 보낼 때 사용자에게 경고합니다. |
| 중복 사용자명 차단 | 사용자명 및 접속 상태 | 동일 이름 사용으로 인한 라우팅 혼동을 방지합니다. |
| 보안 세션 정보 및 fingerprint 표시 | 사용자 확인 | GUI와 명령어를 통해 cipher, session ID, 공개키 fingerprint를 확인할 수 있게 합니다. |
| pytest 기반 프로토콜 검증 | 패킷 처리 로직 | 패킷 packing/unpacking, 크기 제한, sequence 검증, 잘못된 패킷 방어를 테스트합니다. |

## 6. 남아 있는 제한 사항

| 제한 사항 | 설명 |
|---|---|
| 전체 기능에 대한 종단 간 암호화 미보장 | 기본 구조는 클라이언트-서버 간 암호화 채널입니다. 서버는 일반 chat, 일반 whisper, 파일/이미지 라우팅을 위해 각 클라이언트와의 암호화 채널에서 packet을 복호화합니다. |
| E2E 적용 범위 제한 | 실험적 E2E mode는 1:1 텍스트 whisper만 대상으로 하며, 전체 채팅 broadcast와 파일/이미지 전송은 E2E 대상이 아닙니다. |
| E2E metadata 노출 | 서버는 E2E 메시지 본문은 복호화하지 못하지만 송신자, 수신자, 전송 시각, packet size, E2E public key fingerprint 같은 metadata는 볼 수 있습니다. |
| E2E key substitution 가능성 | 서버가 사용자 E2E public key를 배포하므로 악의적인 서버가 key substitution을 시도할 수 있습니다. 사용자 E2E fingerprint TOFU나 key pinning은 아직 적용되지 않았습니다. |
| E2E key 영속화 미적용 | 클라이언트 E2E key는 세션 단위로 생성되므로 재접속 시 fingerprint가 바뀔 수 있습니다. |
| Fingerprint 확인의 한계 | 현재 fingerprint 확인은 사용자가 직접 비교할 수 있게 돕는 UX이며, 인증기관 기반 인증이나 자동 key pinning을 제공하지 않습니다. |
| 서버 인증 부재 | 인증기관(CA) 기반 서버 인증은 없습니다. 사용자는 접속한 서버가 기대한 서버인지 자동으로 검증하지 않습니다. |
| TOFU의 최초 접속 한계 | 최초 접속 시 공격자가 개입하면 잘못된 fingerprint가 신뢰 저장소에 등록될 수 있습니다. |
| 서버 키 영속화 미적용 | 현재 서버 공개키는 연결 단위로 새로 생성되므로, 재접속 시 fingerprint 변경 경고가 자주 발생할 수 있습니다. 운영형 신뢰 모델에는 서버 키 영속화가 필요합니다. |
| 키 핀닝/인증서 검증 미적용 | TOFU 저장소는 인증기관 기반 검증이나 강한 키 핀닝을 대체하지 않습니다. |
| replay 방어 범위 | 동일 암호화 채널 안에서 이전 sequence 재사용은 차단하지만, 재접속 후 이전 세션의 sequence 상태를 이어서 추적하지는 않습니다. |
| 파일 실행 위험 | SHA-256 검증은 전송 무결성을 확인할 뿐, 파일 내용이 안전하다는 의미는 아닙니다. 실행 파일이나 스크립트는 사용자가 별도로 주의해야 합니다. |
| 사용자 인증 부재 | 사용자명은 라우팅 식별자 역할을 하지만, 계정 인증이나 권한 검증 체계는 없습니다. |
| 운영 환경 보안 한계 | 이 프로젝트는 운영 환경용 보안 메신저가 아니라 학습 및 포트폴리오 목적의 보안 네트워크 프로젝트입니다. |

## 7. 향후 개선 방향

| 개선 방향 | 기대 효과 |
|---|---|
| 서버 키 영속화 | 서버 재시작 또는 재접속 후에도 동일 fingerprint를 유지해 TOFU 경고를 의미 있게 만들 수 있습니다. |
| 키 핀닝 또는 인증서 기반 검증 | 최초 접속 신뢰 문제를 줄이고 서버 인증을 강화할 수 있습니다. |
| sequence 검증 로그 고도화 | replay 차단 이벤트를 더 자세히 분석하고 운영 로그 정책과 연결할 수 있습니다. |
| E2E 대상 확장 | 현재는 1:1 텍스트 whisper만 대상으로 하는 E2E 보호를 전체 채팅이나 파일 전송으로 확장하는 구조를 실험할 수 있습니다. |
| 사용자 E2E fingerprint TOFU | 사용자별 E2E fingerprint를 저장하고 변경을 감지해 key substitution 위험을 줄일 수 있습니다. |
| QR fingerprint verification | 사용자끼리 E2E fingerprint를 외부 채널에서 비교할 수 있게 만들어 신뢰 확인을 강화할 수 있습니다. |
| 파일 전송 정책 고도화 | 확장자별 차단 정책, 다운로드 승인 흐름, 저장 위치 정책을 더 세밀하게 만들 수 있습니다. |
| 통합 테스트 고도화 | 장시간 연결, 다중 파일 전송, 장애 복구 같은 더 복잡한 시나리오를 추가할 수 있습니다. |
| 보안 로그와 패킷 검사 도구 추가 | 비정상 패킷, 실패한 복호화, 크기 제한 위반 등을 분석하기 쉽게 만들 수 있습니다. |

## 8. 면접 설명 포인트

| 질문 방향 | 설명 포인트 |
|---|---|
| 왜 자체 패킷 프로토콜을 만들었는가 | 텍스트 메시지와 파일 payload를 같은 네트워크 흐름에서 다루기 위해 length prefix, JSON header, binary payload를 분리했습니다. |
| 어떤 보안 문제를 고려했는가 | 네트워크 도청, payload 변조, replay attack, 비정상 크기 패킷, 파일명 경로 조작, 위험 확장자 파일, 중복 사용자명으로 인한 라우팅 혼동을 고려했습니다. |
| 암호화 구조의 한계는 무엇인가 | 서버가 라우팅을 위해 메시지를 복호화하므로, 현재 구조는 클라이언트-서버 간 암호화 채널이며 서버를 신뢰 경계 안에 둡니다. |
| E2E whisper는 무엇을 보호하는가 | `/e2e`로 보낸 1:1 텍스트 whisper 본문은 수신자 E2E public key로 추가 암호화되어 서버가 본문을 복호화하지 못합니다. 다만 metadata와 key substitution 한계는 남습니다. |
| E2E fingerprint는 어떻게 확인하는가 | `/fingerprint 사용자명` 또는 GUI의 `E2E FP 확인` 버튼으로 상대방의 현재 세션 E2E fingerprint를 확인하고 별도 채널에서 비교할 수 있습니다. |
| fingerprint 표시는 왜 필요한가 | 현재 세션에서 사용 중인 공개키 식별 정보를 사용자에게 보여주고, TOFU 저장소와 비교해 서버 fingerprint 변경을 인지할 수 있게 합니다. |
| 테스트는 무엇을 검증하는가 | 패킷 packing/unpacking, header와 payload 크기 제한, sequence 증가와 replay 차단, 잘못된 패킷 방어, 암호화/복호화, SHA-256 해시 검증을 확인합니다. |
| 다음 보안 개선은 무엇인가 | 서버 키 영속화, 키 핀닝 또는 인증서 검증, 사용자 E2E fingerprint TOFU, E2E 대상 확장, 통합 테스트 고도화를 우선순위로 설명할 수 있습니다. |
